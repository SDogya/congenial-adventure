

### Техническое Задание (ТЗ) для Code Agent: Архитектура ML-Эксперимента

**Цель:** Сгенерировать масштабируемую, модульную инфраструктуру для проведения ML-экспериментов с использованием PyTorch Lightning, Hydra и FSDP, обеспечивающую строгую воспроизводимость и изоляцию компонентов.

#### 1. Топология проекта (Project Structure)
Агент должен создать следующую структуру директорий. Запрещено сваливать все в один файл.
```text
project_root/
├── conf/
│   ├── config.yaml          # Главный файл Hydra
│   ├── model/               # Конфиги моделей (baseline)
│   └── strategy/            # Конфиги FSDP/DDP
├── src/
│   ├── core/
│   │   ├── system.py        # LightningModule (оркестрация шагов)
│   │   └── datamodule.py    # LightningDataModule (I/O данных)
│   └── utils/
│       └── setup.py         # Утилиты детерминизма и логирования
├── run.py                   # Точка входа (Hydra Entrypoint)
└── requirements.txt
```

---

#### 2. Декларативный слой конфигурации (Hydra + Dataclasses)
**Требование:** Отказ от нетипизированных словарей (`dict`) и `argparse`. Конфигурация должна быть строго типизирована.

* **Задача для агента:** Создать файл `src/core/config_schema.py` с использованием `@dataclass`.
* **Спецификация классов:**
    * `FSDPConfig`: параметры шардирования (например, `sharding_strategy: str = "FULL_SHARD"`, `mixed_precision: str = "bf16"`).
    * `ExperimentConfig`: глобальные параметры (`seed: int`, `batch_size: int`, `learning_rate: float`, вложенный `strategy: FSDPConfig`).
* **Регистрация:** В `run.py` использовать `ConfigStore.instance()` для валидации YAML-файлов Hydra против этих Python-датаклассов.

---

#### 3. Оркестратор вычислений (PyTorch Lightning)
**Требование:** Изоляция математики (графов PyTorch) от инженерной логики (циклы, чекпоинты, логирование).

* **Задача для агента:** Реализовать `LitSystem(pl.LightningModule)` в `src/core/system.py`.
* **Контракт класса:**
    * `__init__(self, cfg: ExperimentConfig)`: Принимать строго типизированный конфиг и вызывать `self.save_hyperparameters()`.
    * Внедрить методы `training_step` и `validation_step`, которые возвращают только словари с лоссами и метриками.
    * **Запрет:** Не хардкодить инициализацию моделей внутри `__init__`. Оставить заглушку (плейсхолдер) или паттерн "Фабрика" для будущей инъекции `baseline` модели или `hypothesis` модуля.
* **Оптимизация:** Настроить `configure_optimizers`, используя `torch.optim.AdamW` и косинусное расписание (`CosineAnnealingLR`).

---

#### 4. Распределенное шардирование и I/O (FSDP & Checkpointing)
**Требование:** Подготовка к обучению крупных VLA-моделей без Out-Of-Memory (OOM) ошибок и сетевых заторов.

* **Задача для агента:** В точке входа `run.py` настроить инстанс `pl.Trainer`.
* **Настройки Trainer:**
    * `strategy="fsdp"` (Fully Sharded Data Parallel) с подтягиванием параметров из `FSDPConfig`.
    * `precision="bf16-mixed"` для ускорения матричных умножений.
* **Чекпоинты:** Настроить `ModelCheckpoint` callback. **Критическое требование:** Включить асинхронное сохранение (если поддерживается версией) или сохранение только с нулевого ранга (Leader Node), чтобы избежать I/O-коллапса (Race conditions) при записи на диск.

---

#### 5. Детерминизм и защита от сбоев (Production Safety)
**Требование:** Устранение "тихого" аппаратного недетерминизма GPU для обеспечения побитовой воспроизводимости (bit-exact reproducibility).

* **Задача для агента:** Создать функцию `enforce_determinism(seed: int)` в `src/utils/setup.py` и вызывать её в самом начале `run.py`.
* **Жесткие директивы (Must Implement):**
    * Установить `pl.seed_everything(seed, workers=True)`.
    * Включить `torch.use_deterministic_algorithms(True)`.
    * Прописать настройку переменных окружения внутри скрипта: `os.environ["CUBLAS_WORKSPACE_CONFIG"] = ":4096:8"` (отключает стохастические эвристики cuDNN).
    * Для защиты от таймаутов коллективных операций настроить переменную `os.environ["NCCL_ASYNC_ERROR_HANDLING"] = "1"`.

---

#### 6. Логирование и метаданные
* **Задача для агента:** Подключить `WandbLogger` (Weights & Biases) в `pl.Trainer`.
* Убедиться, что весь объект `ExperimentConfig` корректно сериализуется и отправляется в дашборд W&B при старте эксперимента.
