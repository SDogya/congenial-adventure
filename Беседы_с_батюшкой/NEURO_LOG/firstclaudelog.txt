❯ Код имеет грамотную архитектуру: корректно реализовано отсоединение           
  градиентов (`detach()`) для Decoupled Training, правильно применяется         
  статическая компиляция графов (`torch.compile`) и логика nested dropout.      
  Однако в реализации присутствуют скрытые баги, проблемы производительности в  
  inference-цикле и хрупкие паттерны, которые сломаются при масштабировании.    
                                                                                
  Ниже представлен детальный разбор узких мест и способы их устранения.         
                                                                                
  ### 1. Критическая проблема с планировщиком `CosineAnnealingLR`               
  В `LitSystem.configure_optimizers` жестко захардкожено значение `T_max=100`.  
  Планировщик достигнет нулевого learning rate на 100-м шаге или эпохе (в       
  зависимости от настройки интервала), что почти гарантированно не совпадет с   
  реальной длительностью обучения, сломав сходимость.                           
                                                                                
  **Решение:** Использовать динамический подсчет шагов из самого Lightning:     
  ```python                                                                     
  def configure_optimizers(self) -> Dict[str, Any]:                             
      optimizer = torch.optim.AdamW(                                            
          self.model.get_optimizer_params(),                                    
          lr=self.cfg.learning_rate,                                            
      )                                                                         
      # Используем общее количество шагов, вычисленное Trainer'ом               
      scheduler = CosineAnnealingLR(optimizer,                                  
  T_max=self.trainer.estimated_stepping_batches)                                
      return {                                                                  
          "optimizer": optimizer,                                               
          "lr_scheduler": {"scheduler": scheduler, "interval": "step"}          
      }                                                                         
  ```                                                                           
                                                                                
  ### 2. Рискованная фильтрация параметров в оптимизаторе                       
  Текущая реализация `get_optimizer_params` фильтрует веса по вхождению строк:  
  `'router' not in n` и `'crh' not in n`. Если внутри `obs_encoder` (или в      
  базовой архитектуре трансформера) появятся слои с такими подстроками в        
  названии, они ошибочно лишатся weight decay и получат заниженный learning     
  rate `1e-4`.                                                                  
                                                                                
  **Решение:** Использовать сравнение по идентификаторам объектов памяти:       
  ```python                                                                     
  def get_optimizer_params(self) -> List[Dict[str, Any]]:                       
      router_crh_params = set(self.router.parameters()) |                       
  set(self.crh.parameters())                                                    
      base_params = [p for p in self.parameters() if p not in                   
  router_crh_params]                                                            
                                                                                
      return [                                                                  
          {"params": base_params},                                              
          {"params": list(router_crh_params), "weight_decay": 0.0, "lr": 1e-4}, 
      ]                                                                         
  ```                                                                           
                                                                                
  ### 3. Производительность в `predict_action` (Отсутствие KV Cache)            
  В методе `predict_action` на каждом шаге `t` формируется массив `tokens_in`   
  длиной $1 + t$, и вся эта последовательность заново прогоняется через         
  `self.ar_model(tokens_in, ...)`. Это приводит к квадратичной сложности на     
  каждом шаге ($O(H_l^3)$ суммарно). Если размер латентного окна $H_l$ станет   
  больше 8-16, это станет серьезным бутылочным горлышком.                       
                                                                                
  **Решение:** В идеале `ARModelWithHiddens` должен поддерживать                
  инкрементальный `forward` с передачей `past_key_values`. Если текущая версия  
  `oat` не позволяет модифицировать трансформер под KV-кэш, добавьте явный      
  комментарий о техническом долге, чтобы не забыть об этом ограничении при      
  увеличении горизонта планирования.                                            
                                                                                
  ### 4. Потенциальный краш `nn.ModuleDict`                                     
  Метод `set_normalizer` пытается добавить нормализатор в `nn.ModuleDict`:      
  ```python                                                                     
  self.normalizer.update({'action': normalizer['action']})                      
  ```                                                                           
  Объект `nn.ModuleDict` строго требует, чтобы все его значения наследовались   
  от `torch.nn.Module`. Если `LinearNormalizer` из пакета `oat` является,       
  например, обычным dataclass'ом или словарем, код выбросит `TypeError`.        
  Убедитесь, что `normalizer['action']` является сабклассом `nn.Module`, либо   
  сохраняйте его статистику вручную через `register_buffer`.                    
                                                                                
  ### 5. Хрупкий `squeeze(1)` для визуальных фичей                              
  Во время forward pass выполняется операция `z_v = z_v.squeeze(1)`. В          
  комментарии указано, что ожидается размерность `[B, 1, D]`. Если в будущем    
  `FusedObservationEncoder` будет настроен на обработку истории (например,      
  `To=3`), этот код сломается.                                                  
                                                                                
  **Решение:** Безопаснее брать последний элемент последовательности или        
  сплющивать тензор:                                                            
  ```python                                                                     
  if z_v.dim() == 3:                                                            
      z_v = z_v[:, -1, :] # Берем фичи из последнего наблюдения                 
  ```                                                                           
                                                                                
  ### 6. Засорение логов в PyTorch Lightning                                    
  В `system.py` при вызове `self.log` передается только `loss`. Поскольку       
  `batch` является словарем, Lightning часто не может автоматически вывести     
  `batch_size`, что приведет к появлению warnings на каждом шаге.               
                                                                                
  **Решение:** Явно указывать размер батча:                                     
  ```python                                                                     
  batch_size = batch['action'].size(0)                                          
  self.log("train_loss", loss, batch_size=batch_size, on_step=True,             
  on_epoch=True, sync_dist=True)                                                
  ```                                                                           
                                                                                
  ### 7. Мелкие недочеты                                                        
  * **Нули в латентах при Early Exit:** В `predict_action` при паддинге         
  латентов:                                                                     
    ```python                                                                   
    torch.zeros(B, pad_size, self.embedding_dim, device=obs.device,             
  dtype=latents.dtype)                                                          
    ```                                                                         
    Использование чисто нулевого эмбеддинга для паддинга безопасно только в том 
   случае, если `decode_coarse` игнорирует эти позиции, либо если нулевой       
  вектор семантически безопасен для декодера. Обычно в таких архитектурах       
  используется специальный learnable PAD-эмбеддинг.                             
  * **Autocast Device:** Контекст `with                                         
  torch.autocast(device_type=logits.device.type, enabled=False):` отработает    
  без сбоев на GPU, но если `logits.device.type` вернет нестандартную строку,   
  это может выдать ошибку. Каст `view(-1).float()` внутри самого                
  `F.binary_cross_entropy_with_logits` корректен и предотвращает расходимость в 
   BF16.
